### Phase E — Investigating Telegram polling conflicts
Problem observed in logs:
- `telegram.error.Conflict: terminated by other getUpdates request`

Investigation steps:
1. Inspect runtime logs.
2. Enumerate Python processes on Windows.
3. Inspect `taxsentry-core/src/taxsentry/bot/telegram_bot.py` and runtime bootstrap behavior.
4. Compare development-tree behavior with installed/runtime paths under `C:\Users\Admin\.taxsentry`.
5. Determine whether the conflict came from bot logic or duplicate runtimes.

Key finding:
- Two bot processes were polling the same Telegram bot concurrently.
- They came from two separate Python runtimes:
  1. `D:\TaxSentry\.venv\Scripts\python.exe -m taxsentry.bot.telegram_bot --admin-chat-id 7150649527`
  2. `C:\Users\Admin\AppData\Roaming\uv\python\cpython-3.12.8-windows-x86_64-none\python.exe -m taxsentry.bot.telegram_bot --admin-chat-id 7150649527`

Conclusion:
- The polling conflict was caused by duplicate bot instances, not by a random Telegram/network issue.
- PID-file-only tracking was insufficient to observe or clean up the full runtime state.

### Phase F — Upgrading bot cleanup from PID-only tracking to real process scanning
Main work:
- Reworked launcher logic to detect processes via command-line marker (`taxsentry.bot.telegram_bot`).
- Combined process scan data with PID-file tracking.
- Added stale PID cleanup behavior.
- Updated `status` to show multiple matching PIDs when duplicate runtimes exist.
- Updated `service logs` to surface a clearer warning when Telegram conflict is detected.

Real verification:
1. Start bot.
2. Run `status`.
3. Confirm multiple PIDs are reported.
4. Run `stop`.
5. Confirm signals are sent to all matching bot processes.
6. Run `status` again.
7. Confirm bot is no longer running.

Outcome:
- The new cleanup flow successfully handled duplicate bot runtimes.
- This materially improves local runtime stability, especially on machines with multiple Python interpreter paths.

## Technical decisions reinforced by this work
1. Keep service management cross-platform instead of building a Windows-only path.
2. Separate:
   - artifact generation
   - OS registration / control
3. Do not rely solely on PID files for bot runtime management.
4. Use command-marker-based process scanning to detect duplicate runtimes.
5. Treat Task Scheduler `Access is denied` as a permission boundary of the current terminal session, not as proof of broken generator logic.
6. Treat `telegram.error.Conflict` primarily as a duplicate-runtime signal before suspecting bot polling logic.
